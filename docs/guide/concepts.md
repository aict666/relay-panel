# 核心概念

理解下面几个对象后，RelayPanel 的配置会简单很多。

## 面板与节点

**relay-panel** 是控制面：保存用户、套餐、设备分组和转发规则，通过 Web UI 管理配置。**relay-node** 是数据面：监听端口并真正转发 TCP/UDP 流量。

节点通过 HTTP 上报状态，并通过 WebSocket 接收即时配置；WebSocket 不可用时仍会按轮询周期拉取配置。因此“WebSocket 断开”通常意味着配置生效变慢，不等于现有转发立即中断。

## 设备分组

设备分组是一条线路的逻辑容器，包含一个或多个 relay-node，并提供：

- 节点认证 token。
- 可用于规则监听和多跳链的端口池。
- 线路名称、类型和计费倍率。
- 普通用户是否可见等展示设置。

同组内有多个节点时，每台节点会接收该分组对应的规则。云安全组和主机防火墙应对这些节点保持一致。

## 转发规则

一条规则由以下部分组成：

- **入口分组与入口端口**：客户端连接的位置。
- **协议**：`tcp`、`udp` 或 `tcp_udp`。
- **目标**：最终业务地址，或多目标列表。
- **目标选择策略**：固定、轮询、故障转移、权重、最小延迟或最少连接。
- **多跳链**：流量经过的额外设备分组。
- **限制**：带宽、流量、启停状态等。

规则保存后由面板生成各节点所需的 listener 和下一跳配置。修改链路时，未变化的 hop 会尽量复用原端口；显式“重启规则”则会关闭活跃 TCP 连接。

## 直连与多跳

直连规则只有入口节点和最终目标：

```text
客户端 → 入口节点 → 目标
```

多跳规则在入口与目标间增加一个或多个中继分组：

```text
客户端 → 入口节点 → 中继节点 → 出口节点 → 目标
```

多跳能改变网络路径，但每一跳也会引入延迟、端口与防火墙要求。先验证直连，再逐跳增加中继，排障会容易得多。

## TCP、UDP 与 tcp_udp

| 类型 | 适合场景 | 连接或会话特征 |
|---|---|---|
| `tcp` | Web、SSH、自定义长连接 | 有连接状态，支持 keepalive 和节点间 TFO |
| `udp` | DNS、游戏、语音、QUIC | 按来源维护会话，超时后重建 |
| `tcp_udp` | 同端口同时提供 TCP 和 UDP | 两条数据路径并存，UDP 多跳可走独立 UOT 端口 |

UOT 不是一种新的入口协议，而是 UDP 在节点之间的传输方式。TCP 0-RTT 也只优化节点间 hop，不会消除普通客户端到入口的三次握手。完整边界见[高级路由、UOT 与 0-RTT](/ADVANCED-ROUTING-UOT)。

## 套餐、授权与计费

套餐用于给普通用户分配可用线路、规则数、流量和有效期。规则流量按：

```text
(上行 + 下行) × 线路倍率
```

从套餐额度扣除。个人自用可以只保留管理员账号；准备商业化时，再启用用户、套餐商城、余额和订单功能。
