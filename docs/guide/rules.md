# 创建转发规则

## 选择协议

- 业务只监听 TCP：选择 `tcp`。
- 业务只监听 UDP：选择 `udp`。
- 同一端口需要 TCP 和 UDP：选择 `tcp_udp`。

不要因为“不确定”就默认使用 `tcp_udp`。多开放一种协议意味着额外的监听器、防火墙规则和排障面。

## 填写目标

目标使用 `主机:端口`。主机可以是 IP，也可以是域名；使用域名时节点会缓存解析结果并在缓存过期后跟随 DDNS 变化。

多目标规则支持六种选择策略：

| 策略 | 行为 | 推荐场景 |
|---|---|---|
| 固定首目标 `first` | 始终优先第一个目标 | 单目标或严格固定出口 |
| 轮询 `round_robin` | 新连接/会话依次分配 | 容量相近的多个目标 |
| 故障转移 `failover` | 按顺序尝试，失败再切换 | 主备架构 |
| 权重 `weighted` | 按 1–100 权重比例分配 | 目标容量不同 |
| 最小延迟 `least_latency` | 按探测延迟与失败率评分 | 链路质量波动明显 |
| 最少连接 `least_connections` | 选活跃连接/会话最少者 | 长连接或会话时长差异大 |

节点每 10 秒主动探测目标。连续失败或高失败率会触发 30 秒熔断；所有目标都熔断时会自动试探恢复，避免永久锁死。

::: warning UDP 探测边界
原生 UDP 无法在不知道上层协议的情况下确认目标是否真正应答，因此健康探测只能保证 DNS 可解析等基础条件。对 UDP 业务应额外做真实协议探测。
:::

## 配置多跳链

在规则的 hop 列表中按流量经过顺序添加设备分组。每增加一跳，都要检查：

1. 上一跳能访问下一跳节点的公网或内网地址。
2. 下一跳设备组端口池的 TCP/UDP 入站已放行。
3. 端口池足以分配普通 hop 端口和 UOT 独立隧道端口。
4. 各节点时间和 DNS 工作正常。

建议按以下顺序上线：

1. 先创建入口直达目标的规则并验证。
2. 加一个中继 hop，重新测试 TCP/UDP 实流量。
3. 逐跳增加，出现问题时只回退最后一次修改。

## UOT 与 TCP 0-RTT

当前版本中，`udp` 和 `tcp_udp` 多跳链的 UDP 分量都可使用认证 UOT；`tcp` 和 `tcp_udp` 的 TCP 分量可使用 best-effort TCP Fast Open。两个功能默认启用：

```ini
RELAY_ENABLE_UOT=true
RELAY_ENABLE_TCP_0RTT=true
```

如果内核、sysctl、防火墙或对端不支持 TCP Fast Open，节点会退回普通 TCP，不应导致规则中断。UOT 使用独立 TCP 隧道端口，因此防火墙必须允许设备组端口池内的 TCP 入站。

首次启用前请完整阅读[高级路由、健康探测、UOT 与 TCP 0-RTT](/ADVANCED-ROUTING-UOT)，特别是冷启动、重放风险和滚动升级说明。

## 上线检查

- 从入口节点以外的机器发起测试。
- TCP 同时验证建连、双向数据和长连接保持。
- UDP 使用真实业务客户端测试，不要用一次无应答的 `nc -u` 判断成功与否。
- 多目标策略至少让每个目标各接收一次测试流量。
- 观察节点日志中是否有 listener、认证或连接失败。
- 修改已有生产规则前先导出规则 JSON，保留可快速恢复的版本。
